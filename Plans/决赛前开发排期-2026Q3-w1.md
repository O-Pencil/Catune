**目标**：用 ESLint 把目录边界写进规则，违反即报错。

**步骤**：
1. 在 `.eslintrc.js` 加 `no-restricted-imports` 三组：

**输出文件**：
- `.eslintrc.js`（修改）
- `docs/rn-constraints.md`（新增）

**验收**：
- `npm run lint` 全绿（0 error）

### 7-04 周六 · RN 约束 ② 色值检查（warn-only）

**目标**：禁止 `src/ui/` 下 .tsx 写 `'#xxx'` / `'rgb(...)'` 等色值字面量，但 **warn 而非 error**。

**步骤**：
1. 新建 `scripts/check-no-color-literal.mjs`：
   - 扫描 `src/design/**/*.tsx` 和 `.ts`（跳过 `src/design/theme/colors.ts` / `theme/index.ts`）
   - 检测：正则 `'#[0-9a-fA-F]{3,8}'`、`"#[0-9a-fA-F]{3,8}"`、`rgb\(`、`rgba\(`、`hsl\(`、`hsla\(` 出现在 style 对象或内联样式
   - 输出：文件路径 + 行号 + 色值 + 上下文
   - 退出码：发现违规 = 1（用 `npm run check:colors` 调用），但 `npm run lint` 不调用它
2. 在 `package.json` scripts 加：
   ```json
   "check:colors": "node scripts/check-no-color-literal.mjs",
   "check:ui": "npm run check:colors --silent && npm run check:theme-token-usage"
   ```
3. 跑一次看 baseline 命中数（先记录到 `docs/工程体检-0705.md`）。
4. **不**把 `check:colors` 接进 `npm run lint`（按你的决定：warn-only）。

**输出文件**：
- `scripts/check-no-color-literal.mjs`（新增）
- `package.json`（scripts 段新增 2 行）

**验收**：
- `npm run check:colors` 可独立运行，输出违规列表
- `npm run lint` 不受影响（不调用 colors 检查）

**注意**：现有 baseline 可能很高（App.tsx 就有 `'#FFFFFF'`、`'#141414'`），**不强制 0 命中**，只是建立"应当修"的清单。

### 7-05 周日 · App.tsx 瘦身 + 体检报告

#### 5.1 App.tsx 拆分（核心）

**目标**：347 行 → ≤ 150 行；拆出 `src/ui/hooks/useDashboard.ts`。

**App.tsx 现状分段**（约 130 行业务 + 40 行 AppContent 壳 + 30 行类型/工具）：
- 行 1-30: 头部注释 + imports
- 行 31-50: `buildInitialState` 工具
- 行 51-90: `App()` 主组件前半（字体 + launch 状态 + locale + 全部 `useRef` 初始化 + 6 个 setter 函数）
- 行 91-200: `useEffect`（launch 引导 + 启动数据源 + cleanup）+ locale 切换 effect
- 行 201-260: 渲染（SafeAreaProvider + 三态分支）
- 行 261-347: `AppContent` 子组件

**拆分方案**：

新建 `src/design/hooks/useDashboard.ts`，导出 `useDashboard(locale)` 返回：
```ts
{
  state: DashboardState,        // k
  growth: GrowthState,
  mode: Mode,
  bleStatus, wsStatus, wsSendStatus, wsSendInfo,
  setLocale: (l: Locale) => void,
  actions: {
    useSensor, useMock, useBle, useWs, useWsSend,
    calibrate, pinScenario,
  },
}
```

`useDashboard.ts` 内部职责：
- 持有所有 `useRef`（engine / sensor / ble / ws / mock / advice / growth / reminder / memory）
- 持有 `useState`（k / growth / mode / bleStatus / wsStatus / wsSendStatus / wsSendInfo）
- 实现所有 action 函数
- 跑 `useEffect`（launch 后启动数据源 + cleanup；locale 切换重算）
- 处理 `useSensor` / `useBle` / `useWs` 的失败回退

App.tsx 拆分后只剩：
- imports
- 字体加载 + launch 状态
- 渲染三态（loading / LaunchScreen / LocaleProvider + AppContent）
- `AppContent` 接收 `useDashboard` 返回值传给 `AppShell`

**预估行数**：~140 行。

**输出文件**：
- `src/design/hooks/useDashboard.ts`（新增，约 250 行）
- `App.tsx`（修改，347 → ≤ 150）

**注意**：`useDashboard` 返回的函数引用会因每次 render 变化（如果用 useCallback 不彻底）。传给 `AppShell` 的回调函数**用 `useCallback` 包**，避免子组件无效 re-render。
