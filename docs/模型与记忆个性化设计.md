## 0. 一条不能动的边界（先对齐，避免“为 AI 而 AI”）

PRD §5.10 已定：**姿态分类/打分/提醒触发 = 规则**（确定、0 延迟、可靠底线）；**模型只做规则做不好的事 = 自然语言教练文案 + 视觉评估**。
所以下面谈的“让模型更准”，**不是让模型判断驼背更准**（那是 IMU+规则的事），而是让**教练话术更专业、更个性化、更懂这个用户**。这条边界决定了“微调”要微调什么。

## 1. 基于设备性能的模型推荐（初赛可做，ROI 高）

**目标**：打开/下载页根据手机能力推荐合适模型，避免用户在弱机上下大模型 OOM、在强机上只跑小模型浪费。

**采集的设备信号**（已有一半基础）：
| 信号 | 来源 | 用途 |
| --- | --- | --- |
| 总内存 RAM | `expo-device` `Device.totalMemory` | 主分级依据 |
| CPU 架构 ABI | `Device.supportedCpuArchitectures` | arm64 才允许端侧推理 |
| SME2 / i8mm / dot / fp16 | 已有 `CatuneMnn.getStatus().cpu`（`nativeGetCpuInfoJson`） | 加速能力、是否推荐大模型 |
| 剩余存储 | `expo-file-system` `getFreeDiskStorageAsync` | 下载前校验空间 ≥ 模型体积 |

**分级 → 推荐**（落到 `modelCatalog` 已有的 `tags`）：
| 档位 | 判据 | 推荐 |
| --- | --- | --- |
| 入门 | RAM < 4GB 或 非 arm64 | 仅 Qwen2.5-0.5B；大模型置灰+提示 |
| 主流 | 4–8GB arm64 | 默认 0.5B，1.7B 可选 |
| 高性能 | >8GB 且 i8mm/SME2 | **推荐 1.7B**，标“可启 SME2 加速” |

**落地**：新增 `src/mnn/deviceProfile.ts`（探测 + 分级）+ `recommendModel(profile)`；下载卡显示「为你的设备推荐：__，理由：RAM/加速/空间」。
**收益**：技术维度加分点（“按端侧算力自适应”），且降低现场翻车概率。**依赖**：装 `expo-device`。

## 2. 教练文案“更准确”：微调不是初赛的事

把“准确”拆成三层，按性价比从高到低：

1. **知识表 + 提示工程 + few-shot（初赛做）**
   现有 `engine.ts` 的 `ACTION_TEXT` 查表 + 禁词链已是雏形。扩成**临床向颈椎/胸椎建议库**（每种姿态×严重度×时长 给 2-3 条经物理治疗口径校对的话术），模型 prompt 带 1-2 个 few-shot 例子定调。**几小时即可，覆盖 80% 体感**，0 训练成本，且天然过安全链。
2. **检索增强（复赛）**
   把知识库做成本地小向量库（`hash-embedding` 思路，mem-core 里就有），按当前姿态检索片段塞进 prompt → 文案更专业、可引用。
3. **LoRA 微调 → 转 MNN（复赛，别在初赛碰）**
   - 数据：构造「(角度序列+时长+用户上下文) → 理想教练话术」成对样本（几百条，人工+GPT 造）；颈椎康复语料可挑公开理疗内容清洗。
   - 训练：LLaMA-Factory / Qwen 官方 SFT 出 LoRA，合并后用 MNN 转换工具导出 `.mnn`。
   - 成本：数据+算力+转换+真机验证，**以天计**，不可能 4 天内稳。
   **结论**：初赛=知识表+提示工程；微调写进复赛路线，现场话术用“已构建专业语料、复赛上 LoRA”回答即可。

## 3. 长期记忆与个性化（这是真正的差异化，分两层）

**关键认知：习惯预测 与 教练记忆 是两个不同的活，用不同工具。**

### 3.1 习惯统计层（数值时序，自建轻量，初赛可做 lite 版）
- 持久化姿态事件到本地 FS（`expo-file-system`）：`{ts, posture, severity, durationMin}`。
- 按 **时段(hour) × 星期** 聚合 → 找高风险窗口（“你工作日 15–16 点最易驼背”）。
- **预测式提醒**：在高风险窗口前用 `expo-notifications` 本地通知提前介入。
- 纯统计、无 LLM、确定可靠；`growth.ts` 已在记事件，扩成**持久 + 时段直方图**即可。叙事强：“AI 预测你下午易驼背，提前提醒”。

### 3.2 教练记忆层（语义偏好，用 catui-mem，复赛）
- 让**端侧教练跨会话记住用户**：偏好（“别说教”）、主诉（“早上脖子僵”）、历史结论、有效动作。
- 直接复用工作区的 **`catui-mem`（NanoMemEngine）**：7 类记忆、Stanford 评分、艾宾浩斯遗忘、**heuristic 抽取（可无 LLM 运行）**、PII 过滤——正好是“记住用户习惯”的成熟引擎。
- **接入改造点**：mem-core 持久层是 `node:fs`，RN 没有 → 写一个 **`expo-file-system` 存储适配器**替换 `store.ts` 的 readJson/writeJson；引擎逻辑（评分/抽取/链接/淘汰）是纯 TS 可移植。抽取走 heuristic 或端侧 Qwen。
- **分工建议**：习惯**预测**用 3.1 的数值统计（更适合、可靠）；教练**懂你**用 3.2 的 catui-mem（语义记忆）。别用 catui-mem 去做数值预测，也别用数值统计去存语义偏好。

## 4. 云 + 端分工（对齐 PRD 混合架构）

| 能力 | 端侧（Qwen+MNN，实时/离线） | 云端（Qwen-VL/Plus，低频/深度） |
| --- | --- | --- |
| 姿态分类/提醒 | 规则（不上模型） | — |
| 即时教练文案 | ✅ 0.5B 实时生成 + 规则兜底 | — |
| 视觉体态评估 | — | ✅ 侧身照 → 结构化观察（头前倾/圆肩/骨盆） |
| 日报/周报长文 | 短摘要可端侧 | ✅ 长报告润色 |
| 记忆整理/洞察 | heuristic 抽取 | ✅ LLM 抽取/总结（mem-core 的 LLM 路） |

**边界（PRD 硬约束）**：核心闭环 100% 离线可用；云端只增强、可降级到预置；现场 demo 不依赖网络。

## 7. 微调数据如何运用（数据流 + 两个对齐前提）

**数据流**：
1. **风格锚** `training/data/seed_gold.jsonl`（18 条手写，定义 ≤30字+温和+具体动作+`[动作:xxx]`）
2. → `gen_dataset.py` 扩样 → `train.jsonl`(378) / `val.jsonl`(40)，alpaca 格式（已入 git）
3. → `train_sft.py`（transformers+peft，LoRA rank16，**无需 LLaMA-Factory**）→ LoRA；`compare.py` 基线 vs 微调对比
4. → 合并 → MNN 转换（`training/README.md`）→ `.mnn` → 设备设 active
5. → **App 消费**：`adviceOrchestrator` 用同款 INSTRUCTION 发 prompt → 模型输出「≤30字教练话 + `[动作:xxx]`」→ `parseActionTag` 抽标签（干净正文给用户、动作驱动猫点位高亮+「去跟练」）→ `memory.inject` 个性化前缀。

**两个"数据真正用好"的对齐前提（待办，否则微调白练）**：
- **A. 推理 prompt ≈ 训练 prompt**：训练 `input` 是「姿态：头前倾（TECH_NECK）；颈前倾约25°；已持续12分钟。」，而 App `buildPrompt` 现在是「用户现在头部前倾（颈25° 胸X 腰Y）」。**两者格式不一致会削弱迁移**。要么把 `buildPrompt` 的姿态上下文改成与训练 `input` 同格式，要么给训练集补 `buildPrompt` 格式的样本。**二选一，必须做。**
- **B. 记忆前缀进训练**：`buildPrompt` 已注入「已知用户：…」记忆前缀，但训练数据**不含**该前缀。可在 `gen_dataset.py` 扩样时混入带记忆前缀的样本，让微调模型学会"参考已知用户"，记忆个性化才会被模型真正吃进去。
